Untuk SQL Injection, versi rentan terjadi ketika aplikasi membentuk perintah SQL dengan langsung memasukkan input pengguna ke dalam string query sehingga penyerang bisa menyisipkan fragmen SQL yang mengubah logika query; akibatnya penyerang dapat membypass autentikasi, membaca, mengubah, atau menghapus data. Versi aman memisahkan data dari perintah SQL menggunakan prepared statements atau parameterized queries sehingga input diperlakukan sebagai data, bukan kode, dan kombinasi ini dengan penyimpanan password yang di‑hash serta pembatasan hak pengguna basis data secara signifikan mengurangi risiko.

Untuk Cross‑Site Scripting (XSS), versi rentan menampilkan atau menginjeksi input pengguna ke dalam halaman web tanpa melakukan pen‑encoding atau escaping sesuai konteks sehingga skrip berbahaya dapat dieksekusi di browser korban dan menyebabkan pencurian sesi, pengubahan tampilan halaman, atau aksi atas nama pengguna. Versi aman selalu melakukan output encoding berdasarkan konteks tampilan, menggunakan templating yang otomatis melakukan escaping, menghindari penggunaan insertion API yang berbahaya pada DOM, serta menerapkan kebijakan seperti HttpOnly pada cookie dan Content Security Policy untuk membatasi sumber skrip sehingga even jika ada input berbahaya, skrip tidak akan dieksekusi.

Untuk kerentanan upload file, versi rentan membiarkan pengguna mengunggah file apa pun dan menyimpannya langsung di lokasi publik dengan nama asli sehingga penyerang bisa mengunggah web shell atau skrip yang dapat dieksekusi oleh server dan menyebabkan remote code execution atau penyebaran malware. Versi aman melakukan validasi ketat pada sisi server terhadap ekstensi, MIME type, dan signature/magic bytes file, membatasi ukuran, menyimpan file di lokasi non‑public dengan nama opaque seperti UUID, menjalankan pemindaian malware sebelum file tersedia, dan hanya menyediakan akses melalui mekanisme terkontrol seperti signed URL sehingga file tidak dapat langsung dieksekusi atau diakses tanpa otorisasi.

Untuk Broken Access Control, versi rentan hanya mengandalkan pemeriksaan pada sisi klien atau mengizinkan akses ke resource berdasarkan parameter yang mudah diubah tanpa verifikasi server‑side sehingga pengguna bisa mengakses atau memodifikasi data milik pengguna lain atau mengakses fungsi admin. Versi aman selalu menerapkan pengecekan otorisasi di sisi server pada setiap endpoint dan pada tingkat objek, memastikan bahwa setiap permintaan diperiksa apakah peminta memang pemilik resource atau memiliki peran yang sesuai, menggunakan deny‑by‑default dan logging untuk mendeteksi anomali, serta memadukan penggunaan identifier opaque seperti UUID dan token temporal untuk membuat tebakan resource menjadi sulit.
